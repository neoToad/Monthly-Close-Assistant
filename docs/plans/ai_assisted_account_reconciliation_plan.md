# Implementation Plan: AI-Assisted Account Reconciliation & QuickBooks Fix Writes

**Status:** Draft — awaiting approval.

## Goal

Turn the bank-balance panel from a read-only gap display into an interactive
reconciliation workflow. For each unreconciled cash account, the assistant should:

1. Propose specific, explainable fixes for the bank-vs-GL gap.
2. Show a preview of each proposed QuickBooks write.
3. Let the user confirm individual suggestions.
4. Execute confirmed suggestions as **real QuickBooks writes** via the
   python-quickbooks SDK.
5. Re-check the account and refresh the dashboard.

This addresses the current sandbox gap where the bank ending balance is
**–$3,621.93** and the posted GL total is **–$568.38**.

## Design decisions

1. **Fix suggestions come from an LLM over structured inputs.** We already have a
   deterministic/Claude agent path in `core/agent/summary.py`; this work adds a
   second agent in `core/agent/reconcile.py` specialized for account-level fixes.
2. **No auto-application.** Every suggested write requires explicit user
   confirmation. The default UI is a dry-run preview.
3. **Only create adjusting entries.** The assistant never deletes or edits
   existing QuickBooks objects. It creates `JournalEntry`, `Deposit`, or
   `Purchase` rows as needed.
4. **Use the existing `BankStatementBalance` as the control amount.** Gaps are
   computed against the stored statement balance, not the live QB
   `CurrentBalance`. In sandbox the statement balance can be auto-seeded from QB,
   but it remains editable.
5. **Audit trail in local `Flag` rows.** After a fix is applied, the assistant
   writes a `BALANCE_RECONCILIATION` flag (or updates the existing one) noting
   what was created and the remaining gap.
6. **Production safety.** The command/view layer requires a confirmation token
   (e.g. `dry_run=false`) before calling QB. We add a visible sandbox/production
   badge in the UI.

## User-facing flow

### Dashboard panel

Each unreconciled row in the Bank Balances panel gets a **Reconcile** button.

```
Operating Checking
Bank ending: -$3,621.93 · Posted GL: -$568.38 · Difference: -$3,053.55
[Reconcile]
```

### Reconcile modal

Clicking **Reconcile** opens a modal (`#reconcile-account-modal`) containing:

- **Account summary** (left)
  - Stored statement balance
  - QB CurrentBalance (reference only)
  - Posted GL total
  - Difference
- **Unmatched items** (collapsible)
  - GL rows with no bank match
  - Bank-only rows with no GL match
  - Matched pairs with amount/date differences
- **Suggested fixes** (right)
  - Cards generated by `core/agent/reconcile.py`.
  - Each card shows: description, amount, confidence, QB object type, and an
    **Apply** checkbox.
- **Manual actions**
  - Create a custom journal entry with pre-filled residual amount.
  - Update the stored statement balance.
  - Mark the remaining gap as a known timing item.

### Preview → confirm → execute

1. User checks one or more suggestions and clicks **Preview selected**.
2. Modal swaps to a preview pane listing exact QuickBooks objects that will be
   created:
   - `JournalEntry: debit Bank Fees $53.55, credit Operating Checking $53.55`
   - `Purchase: ACH Transfer $3,000.00 from Operating Checking`
3. User clicks **Apply to QuickBooks**.
4. Backend writes to QB, re-runs `run_reconciliation(month, realm_id)`, and
   returns the updated bank balances section via HTMX.

## Data model changes

### New: `AccountReconciliationState`

Track progress per `(company, qb_account_id, month)` so the workflow is
resumable and auditable.

```text
company           FK → QuickBooksCompany   CASCADE
qb_account_id     CharField(50)
month             CharField(7)            YYYY-MM
statement_balance DecimalField(12,2)
posted_total      DecimalField(12,2)
difference        DecimalField(12,2)
status            CharField               choices:
                    unreconciled, in_progress, reconciled, reviewed
reviewer_notes    TextField               optional
last_suggestions  JSONField               cache of latest LLM suggestions
updated_at        DateTimeField           auto_now
unique_together   (company, qb_account_id, month)
```

This table is optional for the MVP but recommended if we want to show a
reconciliation status history.

### `Flag` additions (optional)

Add a `notes` or `audit_json` field to `Flag` to store what QB objects were
created. If we want to avoid a migration, we can append that information to the
`reason` text instead.

## Backend design

### New module: `core/agent/reconcile.py`

```text
suggest_account_fixes(
    month: str,
    realm_id: str,
    qb_account_id: str,
    qb_api_client: Any | None = None,
    llm: Any | None = None,
) -> dict
```

Returns:

```json
{
  "suggestions": [
    {
      "id": "sug-1",
      "type": "purchase",
      "description": "Bank-only ACH Transfer for $3,000.00 has no GL match",
      "amount": "3000.00",
      "date": "2026-06-12",
      "vendor": "ACH Transfer",
      "account_id": "qb-acc-1",
      "confidence": "high",
      "lines": [
        {"account_name": "Operating Checking", "amount": "-3000.00", "posting": "Credit"},
        {"account_name": "Uncategorized Expense", "amount": "3000.00", "posting": "Debit"}
      ]
    },
    {
      "id": "sug-2",
      "type": "journal_entry",
      "description": "Residual $53.55 likely bank fees / rounding",
      "amount": "53.55",
      "date": "2026-06-30",
      "confidence": "medium",
      "lines": [
        {"account_name": "Bank Fees", "amount": "53.55", "posting": "Debit"},
        {"account_name": "Operating Checking", "amount": "-53.55", "posting": "Credit"}
      ]
    }
  ]
}
```

Implementation:

1. Load inputs:
   - `BankStatementBalance` for the account/month.
   - Posted GL total and the `Transaction` rows contributing to it.
   - Unmatched `BankTransaction` rows.
   - Unmatched `Transaction` rows.
   - QB `CurrentBalance` (optional reference).
   - Prior-month ending balance from `BankStatementBalance` (optional).
2. Build a prompt for the LLM using `build_account_reconcile_prompt()`.
3. Parse the response with a Pydantic model (or simple JSON validation).
4. Fall back to deterministic suggestions when no API key is configured:
   - Residual difference → journal entry against a default adjustment account.
   - Bank-only rows → purchase/expense suggestions.
   - Unmatched GL rows → flag as missing from bank feed.

### New module: `core/quickbooks/writes.py`

Thin wrappers around python-quickbooks object creation. Each function logs and
returns the created object id.

```text
create_journal_entry(
    qb_client: QuickBooks,
    lines: list[dict],
    txn_date: str,
    private_note: str,
) -> dict

create_purchase(
    qb_client: QuickBooks,
    vendor_name: str,
    amount: Decimal,
    account_id: str,
    txn_date: str,
    category_account: str,
) -> dict

create_deposit(
    qb_client: QuickBooks,
    amount: Decimal,
    account_id: str,
    txn_date: str,
    category_account: str,
) -> dict
```

These functions map account names to QB `AccountRef` ids by querying
`QBAccount` rows. They never hit QB for account lookup more than once per call.

### New view: `reconcile_account` (or `suggest_account_fixes`)

Two-step POST endpoints:

1. **GET/POST `dashboard/account/<qb_account_id>/suggest/`**
   - Returns the reconcile modal HTML with suggestions.
2. **POST `dashboard/account/<qb_account_id>/apply/`**
   - Accepts a list of selected suggestion ids, `month`, `realm_id`, and
     `dry_run`.
   - On dry run: returns a preview of objects to create.
   - On confirmation: calls `core/quickbooks/writes.py`, runs
     `run_reconciliation(month, realm_id)`, and returns the updated bank
     balances section.

### Update `run_reconciliation`

After applying fixes, re-running reconciliation should:

1. Sync the new QB transactions into local `Transaction` rows (optional; if we
   trust the write, we can short-circuit by also creating the local row).
2. Recompute balance flags.
3. Show the now-smaller (or zero) gap.

For simplicity, the apply view can call `qb_client.sync_transactions(...)` after
writes to pull the new rows into the local database before re-checking.

## Management commands

### `python manage.py suggest_account_fixes YYYY-MM --realm-id <id> --account-id <id>`

Prints proposed fixes to stdout for CLI users.

Options:
- `--apply` — actually execute the highest-confidence suggestion(s).
- `--dry-run` — default true; show preview only.

### `python manage.py apply_account_fix YYYY-MM --realm-id <id> --account-id <id> --suggestion-id <id>`

Applies a single suggestion by id (used by tests and advanced CLI flows).

## Test plan (TDD)

### Model tests

- `AccountReconciliationState` unique-together on `(company, qb_account_id, month)`.
- `AccountReconciliationState.status` defaults to `unreconciled`.

### Agent suggestion tests

- Deterministic fallback returns a journal-entry suggestion for the residual gap.
- Bank-only rows generate purchase/deposit suggestions.
- Matched-but-different rows are described in the prompt but not suggested as
  new QB writes.
- LLM path parses valid JSON suggestions.
- Invalid LLM JSON falls back to deterministic suggestions.

### QB write tests (mocked)

- `create_journal_entry` builds the correct python-quickbooks objects.
- `create_purchase` maps account name to `AccountRef` via `QBAccount`.
- Writes are scoped to the correct realm id.

### View tests

- GET to the modal endpoint returns suggestion cards.
- POST with `dry_run=true` returns a preview and does not call QB.
- POST with `dry_run=false` calls QB, runs reconciliation, and updates the bank
  balances section.
- Missing confirmation is rejected unless `dry_run=false` is explicit.

### Integration test

- Seed a bank balance, create a gap, run suggestions, apply the top suggestion,
  assert the resulting `BankStatementBalance`/flag state reflects a smaller gap.

## Open questions

1. Should we create local `Transaction` rows immediately after a QB write, or
   rely on the user running `sync_quickbooks` again? **Recommendation:** create
   them immediately so the panel updates without an extra sync step.
2. What is the default adjustment account name for residual journal entries?
   **Recommendation:** "Bank Fees" for small residuals, "Uncategorized Expense"
   for larger unknowns, configurable via settings.
3. Should the modal show the LLM reasoning? **Recommendation:** yes, but only a
   one-sentence explanation per suggestion; full reasoning can go in a tooltip.
4. Should we support undo? **Recommendation:** not for MVP. Instead, write an
   audit note to the `Flag` reason and let the user delete the QB object manually
   if needed.
5. Should the assistant create deposits for bank-only income? **Recommendation:**
   yes, but classify them as "Miscellaneous Income" by default and let the user
   change the category before applying.

## Risks and mitigations

| Risk | Mitigation |
|------|------------|
| Wrong LLM suggestion creates bad QB data | Dry-run preview + per-suggestion confirmation; never auto-apply. |
| Account name lookup fails | Validate all `QBAccount` refs exist before any QB write; fail fast. |
| Sandbox vs production confusion | Show environment badge in modal; require explicit confirmation. |
| Duplicate adjusting entries | Idempotency key per suggestion id + month; skip if already applied. |
| LLM unavailable | Deterministic fallback always available; no API key required. |
